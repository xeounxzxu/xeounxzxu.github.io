# 클로드코드 7가지 실전 팁 정리

## 핵심 내용 요약
- 이 영상은 클로드코드를 잘 쓰는 사람들의 공통점을 "질문 잘하기"보다 "작업 흐름 잘 설계하기"에서 찾는다.
- 병렬 작업은 하나의 세션에 모든 일을 몰아넣지 않고, 탐색과 구현과 검증을 분리해 전체 처리 시간을 줄이는 방법이다.
- `CLAUDE.md`는 프로젝트의 규칙과 배경지식을 누적해서 매번 같은 설명을 반복하지 않도록 만드는 장치다.
- 플랜 모드는 작업 전에 범위를 정리하고, 서브 에이전트는 그 계획을 실제 병렬 실행으로 바꾸는 역할을 한다.
- 검증 루프는 AI가 만든 초안을 결과물로 확정하기 전, 빌드와 리뷰와 수정 사이클을 짧게 반복하는 습관이다.
- 커스텀 스킬과 훅 자동화는 반복 작업을 템플릿과 규칙으로 굳혀, 개인 생산성을 시스템 수준으로 끌어올린다.
- 결론적으로 클로드코드는 대화형 도구라기보다 작은 작업 운영체제처럼 써야 성능이 살아난다.

## 내용
이 영상은 약 20분 분량으로, 클로드코드를 잘 쓰는 사람들의 작업 습관을 7개의 실전 포인트로 묶어 설명한다. 영상 제목만 보면 단순 팁 모음처럼 보이지만, 실제 메시지는 훨씬 구조적이다. 좋은 결과를 만드는 핵심은 모델에 질문을 잘 던지는 재치가 아니라, 작업을 쪼개고 문서화하고 검증하는 운영 방식에 있다는 것이다.

### 1. 병렬 작업: 한 세션에 모든 걸 몰아넣지 않기
병렬 작업은 가장 직관적이면서도 체감 효과가 큰 방법이다. 많은 사용자가 하나의 대화창에서 조사, 기획, 구현, 디버깅을 모두 해결하려고 하는데, 이렇게 하면 컨텍스트가 쉽게 오염되고 응답도 길어지며 정확도도 떨어진다.

영상이 강조하는 병렬 작업의 의미는 단순히 창을 여러 개 띄우는 것이 아니다. 역할이 다른 일을 분리하는 데 있다. 예를 들어 한 세션은 레퍼런스 조사, 다른 세션은 초안 작성, 또 다른 세션은 결과 검증만 담당하게 하면 각 세션이 더 선명한 목적을 갖는다. 이런 식으로 나누면 모델이 "지금 무엇을 해야 하는지"를 더 잘 유지한다.

실무에서는 다음과 같은 식으로 적용할 수 있다.

- 하나의 세션에서 요구사항 정리
- 하나의 세션에서 코드 구현
- 하나의 세션에서 테스트 실패 원인 분석
- 마지막 세션에서 변경사항 리뷰

이 방식의 장점은 속도뿐 아니라 품질에도 있다. 서로 다른 결과를 비교할 수 있고, 한 세션의 실수가 전체 흐름을 끌고 가지 않기 때문이다.

### 2. `CLAUDE.md`: 프로젝트의 기억을 외부화하기
두 번째 포인트는 `CLAUDE.md` 같은 프로젝트 문서에 투자하라는 것이다. 이 파일은 단순 메모장이 아니라, AI와 협업할 때 반복적으로 필요한 맥락을 외부화한 운영 문서에 가깝다.

예를 들어 프로젝트 구조, 사용 중인 프레임워크, 코드 스타일, 금지 패턴, 자주 실행하는 명령, 배포 전 체크리스트 등을 이 문서에 넣어두면 이후 작업에서 같은 설명을 반복할 필요가 줄어든다. 사람 입장에서는 귀찮은 문서화처럼 느껴질 수 있지만, AI 협업에서는 이 비용이 매우 빠르게 회수된다.

핵심은 "정보를 많이 넣는 것"보다 "자주 필요한 규칙을 안정적으로 넣는 것"이다. 특히 아래 같은 내용은 효과가 크다.

- 어떤 디렉터리에 어떤 파일이 있는지
- 어떤 명령으로 빌드와 테스트를 하는지
- 수정하면 안 되는 파일이나 영역이 무엇인지
- 커밋 메시지 규칙과 배포 규칙이 무엇인지

결국 `CLAUDE.md`는 모델의 성능을 올리는 마법 문서가 아니라, 프로젝트의 반복 맥락을 정형화해서 실수를 줄이는 문서다.

### 3. 플랜 모드: 바로 생성하지 말고 먼저 분해하기
세 번째는 플랜 모드다. 영상의 흐름상 이 단계는 작업 속도를 늦추기 위한 절차가 아니라, 큰 실수를 줄이기 위한 준비 단계로 제시된다.

플랜 모드의 핵심은 구현 전에 작업을 분해하는 것이다. 무엇을 바꿀지, 어떤 순서로 처리할지, 어디서 실패할 가능성이 높은지 먼저 드러내면 이후 실행이 훨씬 안정된다. 특히 여러 파일이 동시에 얽히는 작업이나, UI와 로직과 데이터 계층이 함께 변하는 작업에서는 계획 없는 생성이 쉽게 회귀를 만든다.

좋은 계획은 거창할 필요가 없다. 다음 정도면 충분하다.

- 어떤 파일을 읽고 어떤 파일을 수정할지
- 먼저 검증해야 할 가정이 무엇인지
- 완료 후 어떤 방식으로 확인할지

이 과정을 거치면 AI가 과도하게 앞서가거나, 사용자의 의도와 다른 방향으로 구현을 밀어붙일 가능성이 줄어든다.

### 4. 서브 에이전트: 실행을 분산하고 메인 흐름은 통합에 집중하기
네 번째는 서브 에이전트 활용이다. 병렬 작업이 큰 개념이라면, 서브 에이전트는 그것을 더 체계적으로 실행하는 방법에 가깝다.

서브 에이전트를 쓰는 이유는 메인 작업 흐름을 가볍게 유지하기 위해서다. 메인 세션은 우선순위를 정하고 결과를 통합하는 데 집중하고, 세부 작업은 별도 실행 단위로 위임하는 구조다. 예를 들어 탐색용 에이전트는 코드 구조를 조사하고, 작업용 에이전트는 한정된 파일만 수정하고, 검토용 에이전트는 위험 요소를 찾도록 역할을 나눌 수 있다.

이 접근이 유효한 이유는 AI가 한 번에 모든 역할을 수행할 때보다, 각 역할이 좁고 명확할 때 결과 품질이 좋아지는 경우가 많기 때문이다. 사람 팀에서 기획자, 개발자, 리뷰어의 역할을 나누는 것과 비슷한 구조라고 볼 수 있다.

다만 전제가 있다. 위임 단위가 명확해야 한다는 점이다. 범위가 애매하면 오히려 충돌과 중복 작업이 늘어난다. 그래서 서브 에이전트는 "많이 쓰는 것"보다 "경계가 선명한 작업에만 쓰는 것"이 중요하다.

### 5. 검증 루프: 초안을 결과물로 착각하지 않기
다섯 번째는 검증 루프다. 영상 전체를 관통하는 가장 중요한 태도라고 해도 과하지 않다. 생성형 AI는 빠르게 초안을 만드는 데 탁월하지만, 그 초안을 바로 정답으로 받아들이면 문제가 시작된다.

검증 루프는 보통 다음 순환을 뜻한다.

- 생성
- 실행 또는 빌드
- 오류 확인
- 수정
- 다시 확인

중요한 것은 이 루프를 짧게 자주 돌리는 것이다. 작업 끝에서 한 번 크게 검증하는 것보다, 중간중간 작은 단위로 확인하는 편이 비용이 적다. 특히 코드 작업에서는 테스트, 빌드, 린트, 화면 확인 같은 검증이 반드시 붙어야 한다.

영상 설명에서 다루는 "실전 중심"이라는 말도 결국 여기에 닿아 있다. 잘 만든 프롬프트보다, 잘 설계된 검증 루프가 더 많은 실패를 막는다.

### 6. 커스텀 스킬: 자주 하는 일을 재사용 가능한 단위로 만들기
여섯 번째는 커스텀 스킬이다. 이 부분은 개인 사용자와 팀 사용자 모두에게 중요한 포인트다. 반복되는 업무를 매번 자연어로 길게 설명하는 대신, 특정 작업 절차를 재사용 가능한 스킬로 묶어두면 일관성이 크게 올라간다.

예를 들어 다음과 같은 작업은 스킬화하기 좋다.

- 유튜브 영상을 블로그 글로 바꾸기
- 면접 로그를 구조화된 피드백으로 바꾸기
- 특정 리포지토리 규칙에 맞춰 코드 리뷰하기
- 릴리즈 전에 체크리스트를 순서대로 수행하기

커스텀 스킬의 장점은 단지 시간을 아끼는 데 있지 않다. 결과물의 형식을 표준화하고, 품질 기준을 반복 가능하게 만든다는 점이 더 중요하다. 잘 만든 스킬은 개인의 노하우를 자동화 가능한 프로세스로 바꾼다.

### 7. 훅 자동화: 사람이 빼먹는 단계를 시스템으로 고정하기
마지막은 훅 자동화다. 훅은 특정 시점에 자동으로 검사를 실행하거나 규칙을 강제하는 장치라고 이해하면 쉽다. 사람이 직접 챙길 때는 자주 빠뜨리는 단계들이 있는데, 훅은 이런 문제를 줄여준다.

예를 들어 다음과 같은 자동화가 가능하다.

- 커밋 전에 빌드가 되는지 검사
- 특정 파일 형식이 맞는지 검사
- 금지된 경로 수정이 있는지 확인
- 작업 완료 전에 요약이나 로그를 남기기

훅 자동화의 본질은 편의 기능이 아니라 안전장치다. 특히 AI와 함께 빠르게 작업할수록, 사람이 놓치기 쉬운 기본 검증은 자동화 쪽으로 밀어 넣는 편이 낫다.

## 왜 이 7가지가 하나의 세트로 작동하는가
이 7가지 팁은 각각 따로 봐도 유용하지만, 함께 묶였을 때 진짜 힘을 발휘한다. 병렬 작업과 서브 에이전트는 속도를 만들고, `CLAUDE.md`와 플랜 모드는 방향을 고정하고, 검증 루프와 훅 자동화는 품질을 지키며, 커스텀 스킬은 반복 작업을 누적 가능한 자산으로 바꾼다.

즉 이 영상은 "클로드코드에서 이런 기능도 쓸 수 있다"는 기능 소개 영상이 아니다. 오히려 AI 도구를 팀 생산성 시스템처럼 다루는 법에 가깝다. 그래서 해커톤 우승자나 도구 총괄자가 비슷한 패턴을 보인다는 제목도 어느 정도 설득력을 가진다. 잘하는 사람들은 결국 같은 원칙으로 일하기 때문이다.

## 바로 적용해 볼 만한 최소 실천안
이 영상을 보고 당장 전부 따라 할 필요는 없다. 오히려 아래 세 가지만 먼저 적용해도 체감 차이가 크다.

1. 현재 프로젝트 루트에 `CLAUDE.md`를 만들고 규칙 5가지만 적는다.
2. 하나의 세션으로 모든 일을 처리하지 말고, 최소 두 개의 역할로 분리한다.
3. 작업이 끝나면 반드시 빌드나 테스트를 붙여 검증 루프를 만든다.

이 세 가지만 해도 "AI에게 시켜본다"는 단계에서 "AI와 작업 시스템을 만든다"는 단계로 바로 넘어갈 수 있다.

참고로 이 글은 영상 제목, 공개된 설명, 챕터 정보, 페이지에서 확인 가능한 메타데이터를 바탕으로 재구성했다. 자동 생성 자막의 본문 전체는 직접 추출하지 못했기 때문에, 세부 사례보다는 영상이 전달하는 구조와 실전 포인트를 중심으로 정리했다.

## 인사이트
- AI 도구의 성능 차이보다 중요한 것은 작업을 설계하는 사람의 운영 습관일 수 있다.
- 많은 사용자가 프롬프트 작성에 과도하게 집중하지만, 실제 병목은 역할 분리와 검증 체계 부재에서 생기는 경우가 많다.
- 개인 사용자도 문서화와 자동화 수준을 조금만 올리면 팀처럼 일할 수 있고, 그 지점에서 생산성 차이가 크게 벌어진다.
